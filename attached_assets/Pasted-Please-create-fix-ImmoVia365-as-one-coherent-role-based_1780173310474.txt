Please create/fix ImmoVia365 as one coherent role-based marketplace platform.

Do not randomly redesign the dashboards.
Do not rebuild the dashboard UI.
Keep the existing dashboard design/layout as it is.

The goal is to make the platform logically connected, role-based, and professional.

Main concept:
Before login = public website with limited access
After login = full role-based platform experience

Roles:

* Admin
* Client / Kunde / Project Poster
* Service Provider / Dienstleister

Default language: German
Main market: Switzerland
German must always be complete and default.

---

## 1. Public website before login

Before login, the website should stay mostly like it is now.

Public users can see:

* Home
* So funktioniert ImmoVia365
* Dienstleistungen / categories
* Blog
* Limited previews of projects or service providers
* Login/Register buttons

Public users should understand the platform but should not have full access.

Public users should NOT be able to:

* Contact clients
* Contact service providers
* Send messages
* Unlock projects
* Apply to projects
* See full contact details
* Use dashboards
* Search or browse unlimited data
* Access full project/provider details without registration

Add access limits before registration.

If a public user tries to do more, show a registration prompt.

German examples:

* “Bitte registrieren Sie sich, um weitere Projekte zu sehen.”
* “Erstellen Sie ein kostenloses Konto, um mehr Details zu sehen.”
* “Melden Sie sich an, um Dienstleister zu kontaktieren.”
* “Registrieren Sie sich, um Projekte vollständig zu sehen.”

Main rule:
Public website = preview + marketing + limited browsing
Registered user = full role-based platform access

---

## 11. Main data logic

Keep the system simple.

Main concepts:

1. User Account
2. Client profile if user is Client
3. Service Provider profile if user is Service Provider
4. Projects
5. Project interactions/messages/unlocks
6. Admin management

Service Provider registers → creates/edits service provider profile → sees projects

## 12. Project must be central

Project should be the central object of the platform.

Every project must belong to one Client/User.

When a Client creates a project:

* Save clientId/userId inside the project.
* Link the project to the Client profile automatically.
* Do not ask the Client to re-enter the same personal data if it already exists in their profile.

Project should include:

* projectId
* clientId / userId
* title
* category
* sub-service
* description
* photos
* location/city/canton
* budget
* timeline
* status
* createdAt
* updatedAt

Client dashboard:

* Show only projects created by that Client.
* Client can edit/delete own projects.
* Client can see status/progress.
* Client can see interested service providers/messages/applications.

Service Provider view:

* Browse published projects.
* See basic project info.
* Customer contact details stay hidden until allowed/unlocked.
* After unlock/contact, provider gets access based on platform rules.

Admin panel:

* Admin sees all projects.
* Admin sees which Client posted each project.
* Admin sees which Service Providers interacted with each project.
* Admin can click Project → Client.
* Admin can click Client → all projects.
* Admin can click Service Provider → contacted/unlocked projects.

## 18. Notification and email logic

Do not redesign dashboards.
Only add logical notifications/status updates.

When something important happens, related users should be notified.

Admin actions:

* Admin approves project → project status updates → Client notified → email sent
* Admin rejects/archives project → Client notified → email sent if needed
* Admin approves Service Provider → Provider notified → email sent
* Admin changes credits/membership → Provider notified

Service Provider actions:

* Provider unlocks/contacts/applies to project → Client notified → email sent
* Provider sends message/application → Client notified
* Provider withdraws interest → Client notified if needed

Client actions:

* Client accepts/selects Service Provider → Provider notified → email sent
* Client rejects Provider → Provider notified if needed
* Client replies to message → Provider notified
* Client marks project completed → Provider and Admin notified if needed

Create a simple notification system if missing.

Notification fields:

* notificationId
* recipientUserId
* type
* title
* message
* relatedProjectId
* relatedProviderId
* read/unread
* createdAt

Use RESEND_API_KEY if available for important emails.
German must be the default email language.

## 19. Ofri-light business model

Use a simplified Ofri-style model for MVP.

Project Poster / Client:

* Posts projects for free.
* Does not pay to create a project.

Service Provider:

* Can browse project previews.
* Must unlock/contact project using credits or membership rules.
* Customer contact details stay locked until allowed/unlocked.

MVP credit logic:

* Small project: 5 credits
* Medium project: 10 credits
* Large project: 20 credits
* Premium project: 30 credits

New Service Provider:

* Gets trial/free credits, for example 20 credits.
* Can test the platform.

Admin:

* Can manually add/remove credits.
* Can see credit balance.
* Can see credit transactions.
* Can see which projects were unlocked.
* Can see which providers contacted which projects.

Limit:

* Maximum 5 service providers can unlock/contact the same project.
* After 5, show “contact limit reached” or similar.
* Admin can change this later.

Payment:
Do not make payment too complicated for MVP.
First create the structure:

* credit balance
* unlock logic
* transaction history
* admin manual credit management

Later Stripe can be added for:

* buying credits
* subscriptions
* invoices
* premium plans
